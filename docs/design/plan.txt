Context
SwimBuddz needs a closed-loop credit system called "Bubbles" (₦100 = 1 Bubble) to unify payments across all services (sessions, academy, store, events, transport). The full design is at docs/design/WALLET_SERVICE_DESIGN.md. This plan implements Phase 1 (Core Wallet MVP) — all 13 tables are created as models, but only Phase 1 business logic is active.
Port note: The design doc says 8011, but that's taken by ai-service. Next available port is 8013.

Step 4: Core Business Logic (services/wallet_ops.py)
create_wallet(db, member_id, member_auth_id)

Check if wallet exists (return existing if so)
Create wallet with balance=0
Auto-grant 10 Bubbles welcome bonus: create PromotionalBubbleGrant + WalletTransaction (type=welcome_bonus, direction=credit)
Update balance to 10, update lifetime_bubbles_received

debit_wallet(db, wallet_id, amount, idempotency_key, ...)

Per design Section 7.1 exactly:

Check idempotency — return existing transaction if key exists
SELECT FOR UPDATE on wallet row
Validate status=active, balance >= amount
Create WalletTransaction record with balance_before/after snapshots
Update wallet.balance and wallet.lifetime_bubbles_spent
Commit atomically



credit_wallet(db, wallet_id, amount, idempotency_key, ...)

Same pattern as debit but adds to balance
Frozen wallets can still receive credits (refunds)
Updates lifetime_bubbles_purchased or lifetime_bubbles_received based on transaction_type

check_balance(db, member_auth_id, required_amount)

Read-only check, returns (sufficient: bool, balance: int)

Error classes: InsufficientBalanceError, WalletFrozenError, WalletNotFoundError — raised as HTTPException with appropriate status codes and user-friendly messages from design Section 17.

Step 5: Topup Service (services/topup_service.py)
initiate_topup(db, member_auth_id, bubbles_amount)

Per design Section 6.2 flow:

Validate wallet exists and is active
Calculate naira = bubbles × 100
Create WalletTopup record (status=pending)
Call payments service: POST /internal/payments/intents with purpose="wallet_topup", amount in kobo
Store paystack_authorization_url and access_code on topup record
Return topup with redirect URL



confirm_topup(db, topup_reference, payment_reference, status)

Called by payments service webhook
Idempotent: skip if already completed
Credit wallet using credit_wallet() with idempotency_key topup-{topup_id}
Update topup status to completed


Step 6: Promotional Service (services/promotional_service.py)
grant_promotional_bubbles(db, wallet_id, amount, grant_type, reason, admin_auth_id, expires_in_days)

Create PromotionalBubbleGrant record
Credit wallet via credit_wallet()
Link transaction_id back to grant

Verification

Service starts: docker compose up wallet-service → health check at localhost:8013/health
Gateway proxy works: curl localhost:8000/api/v1/wallet/me (with auth header) → returns wallet or 404
DB tables exist: All 13 tables + alembic_version_wallet created
Wallet creation: POST to create → wallet with 10 Bubble welcome bonus
Debit/credit: Internal debit → balance decreases; credit → balance increases
Idempotency: Replay same idempotency_key → returns existing transaction, no double-charge
Admin freeze: Freeze wallet → debits fail with "Wallet temporarily suspended"
Topup initiation: POST topup → returns Paystack URL (or mock in dev)
